0.2 — Lifecycle & End-to-End Flow
v1.0 · May 2026

Lifecycle & End-to-End Flow

From the moment a team member speaks in standup to the moment a deployment lands safely in production — every step is automated, governed, and auditable.

01Standup
02Extract
03Register
04PR Gate
05Risk Scan
06Approval
07Remediate
08Deploy
1
Daily Standup (Microsoft Teams)

Team holds standup in Teams. A scheduled job fires after the meeting and calls Microsoft Graph API to fetch the transcript by meeting ID.

2
Claude Extracts Tasks

Transcript sent to Claude with a department-specific system prompt. Claude returns structured JSON tasks — engineer, service, task description, risk category, status.

3
Microsoft Excel Upsert

Tasks written to the department's Microsoft Excel worksheet via MS Graph API. New tasks are inserted; existing task IDs are updated. Workbook URL posted to Teams.

4
PR Checklist Gate

Developer opens a PR → GitHub webhook fires → pending standup tasks injected into PR body as a checklist. GitHub Actions posts a visibility comment showing pending vs. completed items — merge is never blocked.

5
Infrastructure Risk Scan

Claude analyses the PR diff against Kubernetes YAML, Terraform, and Prometheus metrics. Returns risk level, confidence score, and reasoning chain.

6
Teams Approval Card

Adaptive Card posted with predicted failure, confidence, and suggested fix. Human engineer clicks Approve or Reject — AI waits.

7
AI Remediation PR

On approval: Claude creates a remediation branch, commits the safer config, and opens a PR with full evidence linked to the originating standup task.

8
GitOps Deployment & Audit

PR merge triggers ArgoCD sync → Kubernetes rollout. Task marked Done in Microsoft Excel. Full event chain written to SQLite audit log.

1.1 — Layer 1: Standup Intelligence
v1.0 · May 2026

Standup Intelligence

Microsoft Graph API fetches the raw meeting transcript; Claude transforms unstructured conversation into department-specific structured tasks — fully automated, no human transcription required.

1
Scheduled Trigger

A cron job fires after each department's standup slot (configured in TEAMS_MEETING_IDS). The meeting ID is looked up by department key.

2
Graph API Transcript Fetch

GET /communications/callRecords/{id}/sessions?$expand=segments — returns the raw transcript. Auth uses client credentials flow (tenant ID + client secret).

3
Claude Extraction

Transcript + department system prompt sent to claude-sonnet-4-6. Prompt caching enabled on the system prompt (≥ 1024 tokens) for cost efficiency across repeated daily runs.

4
Structured JSON Output

Claude returns a JSON array of tasks. Each task receives a deterministic task_id (DEPT-YYYY-MM-DD-NNN) before being written to Sheets.

Example Output — Security Department

// POST /standup { "department": "security", "transcript": "..." } [ { "task_id": "SEC-2026-05-14-001", "engineer": "Aisha", "control_domain": "Identity & Access Management", "asset": "AD Connect Sync", "task": "Rotate service account credentials before expiry", "severity": "HIGH", "compliance_tag": "ISO27001-A.9.2", "status": "Pending", "standup_date": "2026-05-14" } ]

1.2 — Layer 2: Microsoft Excel Task Registry
v1.0 · May 2026

Microsoft Excel Task Registry

A single Microsoft Excel workbook with one tab per department is the live source of truth for all tasks — visible to every stakeholder, updated within minutes of each standup.

Sheet Structure

ColumnTypeNotes
task_idStringDeterministic: DEPT-YYYY-MM-DD-NNN. Primary key for upserts.
ownerStringName extracted from transcript (engineer / consultant / analyst)
taskStringFull task description as extracted by Claude
statusEnumPending → In Progress → Blocked → Done
standup_dateDateISO 8601 date of the standup meeting
pr_linkedBooleanSet TRUE when a PR references this task_id
pr_urlURLLink to the GitHub PR that covers this task
dept-specific cols—e.g. severity, module, risk_category per department schema

Upsert Logic (MS Graph)

def upsert_tasks(dept: str, tasks: list[dict]) -> str: token = get_graph_token(GRAPH_TENANT_ID, GRAPH_CLIENT_ID, GRAPH_CLIENT_SECRET) headers = {"Authorization": f"Bearer {token}"} ws_url = f"https://graph.microsoft.com/v1.0/drives/{EXCEL_DRIVE_ID}/items/{EXCEL_WORKBOOK_ID}/workbook/worksheets/{dept}" rows = requests.get(f"{ws_url}/usedRange", headers=headers).json()["values"] existing = {r[0]: i for i, r in enumerate(rows[1:], 2)} # skip header row for task in tasks: tid = task["task_id"] if tid in existing: requests.patch(f"{ws_url}/cell(row={existing[tid]},column=1)", headers=headers, json={"values": [row_values(task)]}) else: requests.post(f"{ws_url}/tables/0/rows", headers=headers, json={"values": [row_values(task)]}) return f"https://dewa.sharepoint.com/workbooks/{EXCEL_WORKBOOK_ID}"
i
Status is never auto-regressed. Only a human can move a task from Done back to In Progress.
✓
After each upsert, the Excel workbook URL is posted to the department's Teams channel as a notification.

1.3 — Layer 3: PR Checklist Gate
v1.0 · May 2026

PR Checklist Gate

Every GitHub PR receives an auto-generated checklist from that day's pending standup tasks. A GitHub Actions workflow posts a visibility comment listing any pending items — developers can always merge, but the team stays informed.

Correlation Outcomes

OutcomeMeaningSystem Action
MATCHPR diff directly covers a standup taskTask status updated to In Progress in Sheets
PARTIALPR contains changes not described in any standup taskFlag added to PR; reviewer notified via Teams
UNKNOWNNo standup task references these changes at allWarning comment posted on PR; engineer notified to link or create a task in Excel

GitHub Actions Notification

# .github/workflows/standup-checklist.yml name: Standup Task Checklist Notification on: pull_request: types: [opened, edited, synchronize] jobs: checklist-notify: runs-on: ubuntu-latest steps: - name: Notify pending standup tasks uses: actions/github-script@v7 with: script: | const body = context.payload.pull_request.body || ''; const unchecked = (body.match(/- \[ \]/g) || []).length; const checked = (body.match(/- \[x\]/gi) || []).length; if (unchecked > 0) { await github.rest.issues.createComment({ owner: context.repo.owner, repo: context.repo.repo, issue_number: context.payload.pull_request.number, body: `⚠️ **Standup Reminder** — ${unchecked} task(s) pending, ${checked} completed.\nYou may merge — this is a visibility reminder, not a blocker.` }); core.warning(`${unchecked} standup task(s) still pending. Merge is allowed.`); } else { core.notice('All standup tasks checked ✓'); }

2.1 — Layer 4: Infrastructure Risk Intelligence
v1.0 · May 2026

Infrastructure Risk Intelligence

Claude receives the full engineering context — infra diffs, live metrics, historical incidents — and returns a risk assessment before deployment, not after.

What Claude Analyses

Kubernetes YAML diffResource limits, replica counts, pod affinity, config changes
Terraform / Helm HCLInfrastructure mutations: VPC changes, security groups, scaling policies
Prometheus metricspeak_memory_usage, restart_count, cpu_throttle_pct, latency_p99
Historical incidentsPast OOMKills, CrashLoopBackoffs, scaling failures — retrieved from audit log
Standup task contextThe originating task_id and risk_category from the standup registry

Risk Output Schema

// Claude output — Layer 4 { "task_id": "DEVOPS-2026-05-14-001", "risk": "HIGH", "prediction": "OOMKill likely within 2 hours of deployment", "confidence": 82, "risk_category": "Resource Optimization", "reasoning": [ "Proposed memory limit (512Mi) is 23% below current peak (666Mi)", "Service restarted 4 times last 48h — pattern matches prior OOMKill", "Historical incident similarity score: 0.87" ] }
!
Risk levels: LOW (confidence < 40) · MEDIUM (40–69) · HIGH (70–89) · CRITICAL (≥ 90). CRITICAL findings trigger an immediate Teams alert regardless of PR status.

2.2 — Layers 5 & 6: Remediation & Teams Approval
v1.0 · May 2026

Remediation & Teams Approval

Claude generates a safer configuration alternative with a confidence-weighted risk reduction; a human engineer approves or rejects through a Teams Adaptive Card before any file is changed.

!
Core Safety Principle — AI NEVER deploys directly. The system cannot bypass this rule. No approval card = no remediation branch = no deployment. Always: AI Suggests → Human Approves → GitOps Deploys.

Remediation Output (Layer 5)

// Claude Layer 5 — safer configuration diff { "config_diff": "resources.limits.memory: 512Mi → 768Mi", "rollback_strategy": "Revert branch claude/prevent-oom-DEVOPS-001", "confidence": 91, "explanation": "Increasing limit 50% above peak prevents OOMKill", "risk_reduction": "82% → 19%" }

Teams Adaptive Card (Layer 6)

ResponseSystem Action
APPROVERemediation branch created, config committed, AI PR opened, task marked In Progress in Sheets
REJECTRejection logged to audit trail, original deployment continues, task remains In Progress

2.3 — Layers 7 & 8: Remediation PR & GitOps
v1.0 · May 2026

Remediation PR & GitOps Deployment

On approval, Claude authors a traceable remediation PR with full evidence — then ArgoCD delivers the change to Kubernetes and marks the originating standup task as Done.

AI-Authored PR Structure (Layer 7)

Branch: claude/prevent-oom-DEVOPS-2026-05-14-001 Title: [AI Remediation] Prevent Predicted OOMKill — DEVOPS-001 ## Predicted Risk Service: checkout-service · Risk: HIGH (82% confidence) Prediction: OOMKill within 2h of planned deployment ## Standup Traceability - Originating task: `DEVOPS-2026-05-14-001` ✓ linked - Engineer: John · Standup date: 2026-05-14 ## Configuration Change - resources.limits.memory: 512Mi → 768Mi ## Evidence - Peak memory (48h): 666Mi (exceeds proposed 512Mi by 23%) - Restart events (48h): 4 (pattern matches prior OOMKill) ## Standup Task Checklist - [x] `DEVOPS-001` Reduce memory usage — *John* ✓ addressed by this PR

GitOps Deployment Chain (Layer 8)

1PR Merge
2GH Actions
3ArgoCD Sync
4K8s Rollout
5Task → Done
✓
After successful Kubernetes rollout, the originating standup task is automatically updated to Done in Microsoft Excel and a Teams notification is sent to the department channel with the PR link and deployment summary.
i
PreMortem never bypasses PR review, the approval chain, or the GitOps workflow. Even approved remediations go through full code review before ArgoCD acts.
